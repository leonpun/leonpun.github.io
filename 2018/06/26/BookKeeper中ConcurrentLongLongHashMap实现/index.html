发表于 2018-06-26

为了尽可能的节省内存，在 GitHub 上翻到了 BookKeeper 用到的 Map 和 Koloboke 的 HashIntIntMap ，分析一下它们的源码，看看能不能改一个用 long 同时存储 key 和 value，或者 int-int 型的来用用。

ConcurrentLongLongHashMap

首先，注释中写明它是用线性探查解决冲突的，key 的值必须大于等于 0。

数据结构

首先我们看一下它底层的数据结构Section，继承自StampedLock，对于索引这种读多写少（？思考）的情境下。
table为 long 数组交错存储kv，比如table[0]为key1，table[1]为value1。其他的map都有几个参数在随后的方法中解释。

再看一下它的构造函数，它的容量为传入参数capacity的两倍，并且全为 -1。
